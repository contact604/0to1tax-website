# PG사 매출, 그대로 신고하면 위험합니다

커머스, 플랫폼, SaaS 스타트업이라면 **PG사(결제대행사)를 통한 결제**가 필수입니다.

- 토스페이먼츠, 나이스페이먼츠, KG이니시스, 페이플

하지만 PG사 정산 내역을 그대로 신고하면 **부가세 과다 납부 또는 과소 신고 위험**이 있습니다.

---

## PG 매출의 기본 구조

### 일반적인 PG 정산 흐름

```
고객 결제 100,000원
→ PG사 수수료 3% (3,000원)
→ 우리 회사 입금 97,000원
```

**질문**: 여기서 부가세 신고 매출은 얼마인가?

👉 **답은 "비즈니스 모델에 따라 다름"**

---

## 비즈니스 모델별 매출 구분

### 1. 우리가 직접 판매 (커머스, SaaS)

| 항목 | 금액 |
|------|------|
| 고객 결제액 | 100,000원 |
| 부가세 신고 매출 | **100,000원** |
| PG 수수료 | 3,000원 (비용 처리) |

👉 **PG 수수료는 비용으로 처리, 매출은 결제액 전액**

---

### 2. 중개 플랫폼 (마켓플레이스)

| 항목 | 금액 |
|------|------|
| 고객 결제액 | 100,000원 |
| 판매자 정산액 | 85,000원 |
| 우리 중개수수료 | 15,000원 |
| 부가세 신고 매출 | **15,000원** |

👉 **판매자 정산액은 우리 매출 아님 (예수금 처리)**

---

## 스타트업이 자주 실수하는 지점

### 1. PG 정산 내역 = 매출?

❌ **PG 정산서에 나온 금액을 그대로 매출로 신고**

```
PG 정산 내역: 97,000원 (수수료 차감 후)
잘못된 신고: 매출 97,000원
올바른 신고: 매출 100,000원, 비용 3,000원
```

👉 **수수료 차감 전 금액이 매출**

---

### 2. 판매자 정산액을 우리 매출로 신고

❌ **중개 플랫폼인데 판매자 정산액까지 매출로 신고**

```
고객 결제: 100,000원
판매자 정산: 85,000원
중개수수료: 15,000원

잘못된 신고: 매출 100,000원
올바른 신고: 매출 15,000원 (중개수수료만)
```

👉 **판매자 정산액은 예수금, 우리 매출 아님**

---

### 3. 환불·취소 처리 누락

| 상황 | 처리 방법 |
|------|-----------|
| 당월 결제·당월 환불 | 매출 차감 |
| 전월 결제·당월 환불 | 수정세금계산서 또는 매출 차감 |
| 부분 환불 | 환불 금액만큼 매출 차감 |

👉 **환불은 매출 차감, 미처리 시 과다 납부**

---

## 올바른 PG 매출 관리 방법

### ① 비즈니스 모델 명확화
- 우리가 직접 판매? → 결제액 전액 매출
- 중개 플랫폼? → 중개수수료만 매출
- 혼합형? → 항목별 분리

### ② PG 정산 내역 자동 연동
- 회계 프로그램과 PG사 API 연동
- 수작업 입력 최소화

### ③ 매월 PG 매출 vs 회계 매출 대조
- PG 결제 내역 합계
- 재무제표 매출 금액
- **두 금액이 일치해야 함**

### ④ 환불·취소 즉시 반영
- 환불 발생 시 즉시 회계 처리
- 분기말에 몰아서 처리 금지

## 마지막 정리

PG 매출은 **"입금된 금액 = 매출"이 아닙니다.**

1. **비즈니스 모델 명확화** (직접 판매 vs 중개)
2. **PG 수수료와 매출 분리**
3. **판매자 정산액은 예수금 처리**
4. **환불·취소 즉시 반영**
5. **매월 PG 내역과 회계 대조**
